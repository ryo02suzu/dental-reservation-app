## 1. プロダクト概要

**Dental Flow**（デンタルフロー）は、日本の歯科医院を対象にした**クラウド型 SaaS 予約管理システム**です。

| 項目 | 内容 |
|------|------|
| プロダクト名 | Dental Flow（デンタルフロー） |
| 運営会社 | Sourirette合同会社 |
| 対象市場 | 日本国内の歯科医院（個人・法人・グループ医院） |
| 提供形態 | Web SaaS（マルチテナント）・月額課金 |
| UI言語 | 日本語（全UI） |
| デザイン思想 | SmartHR風のシンプル・プロフェッショナルなデザイン |
| ステータス | β版 商業稼働中 |
| URL | `/` ランディングページ → `/demo-login` でデモ体験可能 |

### プロダクトのコアバリュー

> **歯科医院の電話予約を、終わらせる。**  
> 予約・シフト・勤怠・患者台帳・リコール——ばらばらだった業務を、ひとつのシステムにまとめる。  
> 初期費用ゼロ、パスワード不要、翌日から動く。

## 2. 誰のためのシステムか

### 開発の背景

開発者は歯科大学に在籍しており、歯科医師である親の医院を近くで見ながら育ちました。その医院では今でも、予約は電話で、管理は紙の台帳で行われています。

現場を見て、率直に感じました。**このままでは、機会損失が続くだけだと。**

現在は患者の多くがインターネットで歯科医院を探す時代です。どれだけ技術の高い医院でも、オンライン上に予約手段がなければ、新患の獲得機会を逃し続けます。一方で既存のデジタルシステムは初期費用が高く、月額も高く、現実的に導入できないケースが多い。

「良い医療をしている医院が、システムの問題で患者に届かない」——この構造を変えたいという思いがDental Flowの出発点です。

開発にあたっては、大学の指導教員や現役の歯科医師・歯科衛生士からヒアリングを重ねました。「現場で実際に使われること」を唯一の基準として、設計を行っています。

### 日本の歯科医院が抱える課題

1. **予約管理が電話・Fax頼み** → 受付スタッフに負担が集中、診療時間外に予約が取れない
2. **患者情報が紙・Excelに散在** → 来院履歴・アレルギー・次回リコールを探すのに時間がかかる
3. **シフト管理がLINE・口頭** → 院長がシフト集計に毎月数時間を費やす
4. **無断キャンセル・直前キャンセルが多い** → リマインダーが電話一本しかなく、忘れられやすい
5. **定期検診（リコール）の管理が属人的** → 「そろそろ来るはず」の患者に連絡できていない

### ターゲットセグメント

- **主要ターゲット**: 一般歯科・小児歯科・矯正歯科（院長1名 + スタッフ3〜10名規模）
- **拡張ターゲット**: グループ医院・医療法人（複数クリニックの統括管理）
- **地理的対象**: 日本国内全域（クラウドのためエリア制限なし）

## 3. ユーザー種別と提供する体験

Dental Flow には **3種類のユーザー** が存在します。

### ① 院長・受付スタッフ（管理者ユーザー）

- ログイン: ID/パスワード認証（`/login`）
- アクセス先: `/admin`（管理ダッシュボード）
- できること: 予約の閲覧・作成・変更・キャンセル、患者台帳管理、シフト管理、売上レポート、設定変更

### ② 患者

- ログイン: メールアドレス＋パスワード（任意）／ゲスト予約も可能
- アクセス先: `/book/:slug`（予約フォーム）、`/my-appointments`（マイページ）
- できること: オンライン予約、予約確認・キャンセル、来院履歴確認、問診票記入

### ③ スタッフ（歯科医師・歯科衛生士）

- ログイン: QRコードまたは固有URLトークン（`/staff-login/:token`）
- アクセス先: `/my-schedule`（スタッフマイページ）
- できること: 担当予約一覧、シフト確認・申請、出退勤QRコード打刻

## 4. 機能一覧 — 管理画面

### 4-1. 予約カレンダー（日表示）

- **スタッフ別カラム表示**: 歯科医師・歯科衛生士ごとに縦列に分けて表示
- **時間軸**: 縦軸が時間（30分刻み）、横軸がスタッフ
- **現在時刻ライン**: 赤い線で現在時刻を示す
- **ステータス色分け**: 確認済み（緑）・未確認（黄）・未来（青）・キャンセル（グレー）
- **昼休み表示**: 院の昼休み時間帯にハッチングを表示
- **休診日表示**: 休診設定の曜日・祝日はグレーアウト
- **空きスロットクリック**: クリックで予約作成モーダルが開く（日時・スタッフが自動入力）
- **予約カードクリック**: 予約編集モーダルが開く

### 4-2. 予約カレンダー（週表示）

- 1週間を横軸に並べて表示
- 各日のシフトがあるスタッフのみ表示（シフトなし日は非表示）
- 週単位で前後に移動

### 4-3. 予約作成・編集モーダル

以下の項目を入力・変更できます：

| 項目 | 内容 |
|------|------|
| 患者名 | インクリメンタルサーチで患者DB検索、または新規入力 |
| 電話番号 | 患者DBから自動補完 |
| 担当スタッフ | ドロップダウン（スタッフ一覧から選択） |
| 診療メニュー | メニューDBから選択（所要時間を自動セット） |
| 日時 | 日付・開始時刻・終了時刻 |
| チェア番号 | 何番チェアを使うか |
| ステータス | 予約確認済み / 来院済み / キャンセル / 無断キャンセル |
| 診察メモ | 自由記入 |
| 問診票 | 折り畳み展開で記入内容を確認 |

**リコールプロンプト**: 「来院済み」に変更すると、次回定期検診のリコール案内を送信するかどうかを確認するダイアログが表示される。

### 4-4. 患者台帳

- 患者一覧（氏名カナ検索・絞り込み）
- 患者詳細: 基本情報・来院履歴・アレルギー・服薬情報・医療メモ
- 患者番号（patient_number）の自動付番
- キャンセル回数・無断キャンセル回数の記録
- 次回リコール日の設定・管理
- リコールリスト: 定期検診時期が来た患者の一覧・メール/LINE送信

### 4-5. シフト管理（シフトボード）

- **月表示**: カレンダー形式でスタッフ全員のシフトを一覧
- **週表示**: 週単位でシフトを確認・編集
- **ステータス管理**:
  - `requested`（スタッフが申請中）
  - `approved`（院長が承認済み）
  - `rejected`（却下）
- **シフトパターン**: よく使うシフト時間をパターン登録（例: 9:00〜18:00「通常シフト」）
- **一括承認**: 月単位でまとめて承認

### 4-6. 勤怠管理

- 出勤・退勤・休憩開始・休憩終了の記録
- スタッフの勤怠履歴一覧
- 月次勤怠サマリー（総勤務時間・残業時間）

### 4-7. 診療メニュー管理

- メニューの追加・編集・削除・並び替え
- 各メニュー: 名前・説明・所要時間（分）・料金・対象スタッフ種別
- 診療カテゴリ分類（一般歯科・矯正・ホワイトニング等）

### 4-8. 売上・統計レポート

- 月次売上グラフ
- 診療メニュー別売上内訳
- 新患数・再診数の推移
- キャンセル率推移

### 4-9. リマインダー・通知設定

| チャネル | 詳細 |
|---------|------|
| メール | Resend APIを使用。予約確認・前日リマインダー・リコール案内 |
| LINE | LINE公式アカウント連携（アドオン契約）。フォロー時にユーザーID取得、push通知送信 |
| SMS | SMS通知パック（アドオン契約）。予約確認・リマインダー |

- **自動リマインダー**: 毎朝設定した時刻（例: 9:00）に翌日予約患者へ自動送信
- **リコール自動通知**: 定期検診6ヶ月後を過ぎた患者に案内を自動送信

### 4-10. キャンセル待ちリスト

- 希望日時・メニューを登録してキャンセル待ち
- キャンセル発生時にスタッフが待ちリストを確認して連絡

### 4-11. 設定

| 設定カテゴリ | 設定項目 |
|------------|---------|
| クリニック基本情報 | 医院名・電話・住所・説明文 |
| 営業時間 | 曜日別 開院〜閉院・昼休み設定 |
| 休診日 | 日本の祝日と連動・手動追加 |
| 予約ルール | 予約可能期間（最大60日先まで）・バッファ時間・ダブルブッキング可否・要承認設定 |
| スタッフ管理 | スタッフの追加・編集・ログイントークン発行・時給設定 |
| チェア数 | 使用チェア数（デフォルト5台） |
| カラーテーマ | クリニックのブランドカラー設定 |
| メール設定 | Resend APIキー入力 |
| LINE設定 | Channel Access Token・Channel Secret入力・Webhook URL表示 |
| 問診票 | 問診票の有効化設定 |

## 5. 機能一覧 — 患者マイページ・予約画面

### 5-1. オンライン予約フォーム（`/book/:slug`）

患者がクリニックのURLを通じてアクセスする予約フォーム。

**ステップ構成（ウィザード形式）**:
1. 診療メニュー選択
2. 日付・時間スロット選択（空き枠のみ表示）
3. 担当スタッフ選択（任意）
4. 患者情報入力（氏名・電話・生年月日）
5. 確認・送信

**特徴**:
- **ゲスト予約対応**: アカウント不要で予約可能
- **会員ログイン**: メールアドレス＋パスワードで過去の予約と紐付け
- **リアルタイム空き確認**: 予約が入っている時間は自動でグレーアウト
- **バッファ時間**: 直近N分以内の枠は予約不可（設定で変更可）
- **問診票**: 事前にWeb問診票を記入して送信
- **紹介コード**: 患者が友人を紹介できる紹介制度（referralCode）
- **キャンセル制限**: 30日以内にN回以上キャンセルした患者は予約不可

### 5-2. 患者マイページ（`/my-appointments`）

- 今後の予約一覧（日時・クリニック・担当スタッフ・メニュー）
- 予約のキャンセル（理由入力）
- 来院履歴確認
- 次回予約ボタン（予約フォームへ誘導）

## 6. 機能一覧 — スタッフマイページ

### アクセス方法

スタッフ専用のログインURLトークン（`/staff-login/:token`）またはQRコードでアクセス。パスワード不要。

### 6-1. 担当予約一覧

- 今日・明日・今週の担当予約を表示
- 患者名・時間・診療内容・ステータスを確認
- 患者のアレルギー・医療メモをその場で確認

### 6-2. シフト確認・申請

- 月・週単位で自分のシフトを確認
- シフト申請（希望日時を送信）→ 院長が承認
- 承認済み・申請中・却下をステータスで確認

### 6-3. 出退勤 QR打刻

- スマートフォンのカメラでQRコードをスキャン → 出勤・退勤打刻
- 打刻時刻が勤怠DBに記録

## 9. データモデル（全テーブル）

### clinics（クリニック）
マルチテナントの中心。1レコード = 1医院。

| カラム | 型 | 説明 |
|--------|-----|------|
| id | UUID | プライマリキー |
| name | text | 医院名 |
| slug | text | ユニークなURL識別子（例: `demo`, `sakura-dental`） |
| plan_type | text | プラン種別（free / starter / partner / enterprise） |
| is_active | boolean | アカウント有効フラグ |
| phone / email / address / description | text | 基本情報 |

### staff（スタッフ）

| カラム | 型 | 説明 |
|--------|-----|------|
| id | UUID | プライマリキー |
| clinic_id | UUID | 所属クリニック（外部キー） |
| name | text | 氏名 |
| role | text | 役職（doctor / hygienist / receptionist） |
| job_title | text | 役職名（例: 歯科医師・歯科衛生士） |
| employment_type | text | 雇用形態（fulltime / parttime） |
| show_in_calendar | boolean | カレンダーに表示するか |

## 11. マルチテナント設計

### 設計方針

**Single Database / Row-Level Isolation（共有DB・行レベル分離）**

- 全テナント（クリニック）が同一のPostgreSQLデータベースを共有
- すべてのテーブルに `clinic_id` カラムを持ち、クエリ時に必ず `WHERE clinic_id = '...'` を付与
- クリニックAのデータがクリニックBから見えることはない

### テナント識別方法

1. **管理者セッション**: ログイン後セッションに `user.clinicId` を保存、APIリクエスト時に参照
2. **スタッフセッション**: `staffClinicId` をセッションに保存
3. **公開API**: URLの `:slug` パラメータからクリニックを特定

### テナント作成フロー

1. クリニックが `/signup` からアカウント作成
2. `clinics` テーブルにレコード挿入 + `users` テーブルに管理者ユーザー作成
3. `clinic_id` で紐付いた初期データ（営業時間・設定）を自動生成
4. 選択したアドオンを `clinic_addons` に登録

## 12. 外部連携

### LINE連携

**仕組み**:
1. クリニックがLINE公式アカウントを開設
2. LINE DevelopersコンソールでWebhook URLを `https://[ドメイン]/api/line/:slug/webhook` に設定

## 13. セキュリティ設計

### 認証・セッション

| 項目 | 実装 |
|------|------|

| Cookie設定 | Production: `secure: true`, `sameSite: "none"` |
| セッションシークレット | 環境変数 `SESSION_SECRET` で管理 |

### レート制限

| エンドポイント | 制限 |
|-------------|------|
| 予約作成（公開） | 1IP・15分あたり10回 |
| ログイン（患者） | 1IP・15分あたり5回 |
| 患者登録 | 1IP・1時間あたり3回 |
| パスワードリセット | 1IP・1時間あたり3回 |
| キャンセル（公開） | 1IP・15分あたり5回 |
| 一般公開API | 1IP・15分あたり60回 |

### アクセス制御

- **管理ルート**: `requireAuth` ミドルウェアでセッションチェック
- **スーパー管理者ルート**: `requireSuperAdmin` ミドルウェアで `isSuperAdmin` フラグを確認

LINE からの Webhook リクエストは `x-line-signature` ヘッダーのHMAC-SHA256署名を検証。なりすまし防止。

### 入力バリデーション

- 全リクエストボディをZodスキーマで厳密にバリデーション
- SQLインジェクション: Drizzle ORM のパラメタライズドクエリで防止

## 16. 差別化・競合優位性

競合優位性は以下の優先順位で提案します。提案相手（院長・受付・スタッフ）によって強調するポイントを変えると効果的です。

---

### Before / After ——「導入後の未来」

院長が判断するのは「機能」ではなく「どう変わるか」です。

| | Before（現状） | After（Dental Flow導入後） |
|--|----------------|--------------------------|
| 予約受付 | 電話が鳴りっぱなし、診療中も出なければならない | 予約は自動受付、受付は会計と患者対応に集中できる |
| キャンセル対応 | 無断キャンセルに気づくのは当日 | 前日・当日リマインダーで来院率が向上 |
| シフト管理 | LINEで希望収集 → 院長がExcelで集計（毎月数時間） | スタッフがアプリで申請、院長は承認するだけ |
| スタッフのログイン | パスワードを忘れる・設定が大変 | QRコード一枚で即ログイン |
| 定期検診の案内 | 「そろそろ来るはず」と思いながら連絡できていない | 最終来院から自動でリコール通知を送信 |
| 複数システムの管理 | 予約・患者台帳・勤怠でそれぞれ別ツールを契約 | ひとつのシステムに全部入っている |

---

### ① 一体型である（最強の訴求軸）

**「何個もシステムを契約しなくていい」は、誰にでも刺さるメッセージです。**

既存の予約システムの多くは、予約を取ることしかできません。患者台帳は別のソフト、シフト管理はLINEとExcel、勤怠管理はタイムカードか別ツール——歯科医院の運営は、複数のシステムの継ぎ接ぎで成り立っています。その分、コストも手間も、スタッフが覚えることも、増え続けます。

Dental Flow は、**予約・シフト・勤怠・患者台帳・リコール**を一つのシステムで完結させます。これは競合との比較において最も説明しやすい優位性であり、価格の納得感にも直結します。

| 機能 | 一般的な予約システム | Dental Flow |
|------|-------------------|-------------|
| オンライン予約 | ✓ | ✓ |
| 患者台帳 | △（別システム） | ✓（一体型） |
| シフト申請・承認 | ✗ | ✓ |
| QR打刻・勤怠管理 | ✗ | ✓ |
| リコール（定期検診リマインド） | ✗ | ✓ |
| 歯科向け問診票 | ✗ | ✓ |
| 日本の祝日連動 | △ | ✓ |
| チェア番号管理 | ✗ | ✓ |
| フリガナ管理 | ✗ | ✓ |

---

### ② 導入ハードルの低さ（次点）

**ITが苦手な歯科医院でも、翌日から使い始められる設計です。**

本プロダクトで最も重視した設計方針のひとつが「導入のしやすさ」です。どれだけ優れたシステムでも、実際に使われなければ意味がないと考えているためです。

**スタッフのログイン**については、パスワードを不要にしました。院長がQRコードまたは専用URLを共有するだけで、スタッフは即座に利用を開始できます。歯科衛生士やスタッフがITに慣れていない前提で、「アカウント作成・パスワード設定」という工程自体をなくしています。

**患者向けの予約導線**も同様にシンプルです。クリニック専用の予約URLを1つ発行し、ホームページやLINEに掲載するだけで、その翌日からオンライン予約を受け付けられます。設定はすべて管理画面から行えるため、エンジニアの関与は不要です。

**デモ環境**についても、ログイン不要で管理者・患者・スタッフの3画面を即時体験できる設計にしています。営業の場でその場で見せながら説明できます。

| 競合の課題 | Dental Flow の対応 |
|------------|-------------------|
| 初期設定に数日かかる | URLとQRコードで即日スタート |
| スタッフ全員のアカウント発行が必要 | QR/URLトークンのみ（パスワード不要） |
| エンジニアが必要 | 管理画面のみで全設定が完結 |
| 初期費用が高い | 初期費用ゼロ |

---

### ③ スタッフ体験（院長相手では優先度を下げても可）

**既存システムは管理者しか見ていない。これは断言できます。**

多くの競合システムは「院長・受付が使う管理画面」を中心に設計されており、スタッフ（衛生士・歯科医師）が使う画面は後から付け足したものです。結果として、スタッフは使いにくいシステムを毎日強いられ、院長は使い方を説明し続けることになります。

Dental Flow はスタッフマイページを最初から一等市民として設計しています。担当予約の確認・シフト申請・出退勤打刻をスマートフォン一台で完結できます。院長相手の提案では「スタッフが便利かどうか」より「自分の業務が楽になるか」の方が刺さりやすいため、文脈に応じて強調度を調整してください。

---

### ④ LINE連携（「患者が来なくなることを防ぐ仕組み」として訴求）

**「LINE連携があります」だけでは弱い。実装の質で差がつきます。**

LINE公式アカウントと連携しているシステムは他にも存在しますが、Webhook署名検証・チャンネルシークレットによるセキュリティ対応まで実装しているシステムは少ない。「ちゃんと作ってある」ことが信頼につながります。

訴求のポイントは「患者が来なくなることを防ぐ仕組み全体」として説明することです：

- **予約リマインダー**：前日・当日にLINE／メール／SMS自動送信 → 無断キャンセルを防止
- **リコール通知**：最終来院から指定期間後に自動送信 → 定期検診の離脱を防止
- **マルチチャネル対応**：LINEを使っていない患者にもメール・SMSで届く設計

### 価格の納得感

**月額9,800円を「コストが増える」と感じる場合、比較対象が適切でない可能性があります。**

本プロダクトの価値は、**コスト削減**と**機会損失の防止**にあります。

**無断キャンセル防止による効果**
無断キャンセルが1件発生すると、その時間の売上は失われます。一般的な保険診療で3,000〜8,000円程度、自費診療であればさらに高額です。リマインダーによってこの1件を防ぐだけで、月額費用は回収できます。

**受付業務効率化による効果**
電話での予約対応は1件あたり約5〜10分かかります。仮に1日20件の対応がある場合、最大で約3時間を要します。このうち半分がオンライン予約に移行するだけで、1日あたり約1時間の業務削減となります。時給1,100円のスタッフであれば、月20日で約**22,000円分のコスト削減**に相当します。

**つまり、本プロダクトは「コストが増える」のではなく、「現状の非効率による損失を減らす仕組み」です。**

| コスト削減・損失防止の試算 | 月間効果の目安 |
|--------------------------|-------------|
| 無断キャンセル2件防止 | +6,000〜16,000円 |
| 電話予約の半数をオンライン化 | スタッフ工数▲約22,000円相当 |
| リコールによる休眠患者の再来院 | 1件あたり+数千円〜数万円 |
| **合計（保守的試算）** | **月額費用の3〜5倍以上の効果** |
